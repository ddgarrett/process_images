Here is a concise map of how things work today and a concrete way to add **Show → Duplicate Group** without fighting the existing patterns.

## How “Show…” works today

1. **Menu text** comes from `status_menu.py` `get_show_submenu()`, which emits entries like `f'&All::{c.EVT_SHOW_ALL}'` (label + event string for FreeSimpleGUI).

2. **Handlers** are registered in `pi_config.py`: each “Show …” line is a `PiFilterTable(event, filter)` pair. Those filters are created **once at import** and only implement `test(row)` on a single row.

```93:98:pi_config.py
''' global filter table events '''
PiFilterTable(c.EVT_SHOW_ALL, None)
PiFilterTable(c.EVT_SHOW_TBD, FilterTbd())
PiFilterTable(c.EVT_SHOW_POSSIBLE_DUP, FilterPossibleDup())
PiFilterTable(c.EVT_SHOW_POSSIBLE_GOOD_PLUS, FilterPossibleGoodPlus())
PiFilterTable(c.EVT_SHOW_POSSIBLE_BEST, FilterPossibleBest())
```

3. **Applying a filter** clears the visible table to rows where `filter.filter(c.table._original_rows)` keeps the row (`PiFilterTable.handle_event` in `pi_action.py`).

4. **Selection** for status/map/etc. is **not** global: `StatusMenu(rowget=...)` is constructed per widget (`pi_tree_list.py`, `pi_gallery_elem.py`, `pi_image_elem.py`) with different getters (`get_selected_rows`, `get_rows`, `get_row`).

**Important implication:** “Duplicate Group” depends on **which rows are selected in that pane**. It cannot be a static `PiFilterTable(EVT_..., DuplicateGroupFilter())` registered in `pi_config.py` the same way as `FilterTbd`, unless you teach the handler how to discover selection from `values` alone. The pattern that already matches “needs rowget” is **`PiActionMap`**: unique event per instance, `item()` added to the menu, `handle_event` calls `rowget(values)`.

## Recommended shape

### 1. `pi_filters.py` — `DuplicateGroupFilter` (like `SelectedTreeNodesFilter`)

Add a `Filter` subclass whose constructor takes **everything needed to decide membership** (no `values` inside the filter):

- **`all_rows`**: use `c.table._original_rows` when building the filter (same source as `Table.filter_rows`; the project already references `_original_rows` elsewhere, e.g. save in `pi_action.py`).
- **`selected_rows`**: ordered list from `rowget(values)` in the action.

**Anchor (first selected row that qualifies):**

- Walk `selected_rows` in order.
- If `dup_photo` is non-blank → that row is the anchor; **canonical basename** `dup_target = anchor['dup_photo']`.
- Else scan `all_rows` for any row with `dup_photo == anchor['file_name']` → anchor is “canonical side”; `dup_target = anchor['file_name']`.

**Group membership** (your stated rule, interpreted consistently):

- Every row with `file_name == dup_target` (same string as stored in `dup_photo`).
- Every row with `dup_photo == dup_target`.

That includes the anchor in both cases (either it points at `dup_target`, or it is the row named `dup_target`). Implement `test(row)` as `row in group_set` or compare `file_name` / `dup_photo` to `dup_target` with the same equality rules you use for the anchor scan.

**Edge cases to decide in code:**

- If no anchor qualifies, do not change the filter (or call `filter_rows(None)`); update status with a short message.
- If multiple rows share `file_name` across folders, `file_name`-only linking may pull more than one “canonical” row; that matches storing only a filename in `dup_photo`.

### 2. New action (e.g. `PiShowDuplicateGroup` in `pi_action.py` or a small new module)

Mirror `PiActionMap`:

- `__init__(self, rowget=...)`, `next_id`, `_get_event()`, `item()`, `handle_event`.
- In `handle_event`: `selected = rowget(values)`; build `DuplicateGroupFilter(...)` or a plain `set` of rows; `c.table.filter_rows(that_filter)`; `c.listeners.notify(c.EVT_TABLE_LOAD, values)`; `c.update_status(...)`.

Do **not** add another `PiFilterTable` line in `pi_config.py` for this unless you merge duplicate-group logic into a single global handler that knows how to read tree/gallery/image selection from `values` (more brittle).

### 3. Menus — wire like Map, not like `EVT_SHOW_TBD`

Where you build the right-click menu, append one dynamic item to the Show submenu, same tier as existing entries:

- `pi_tree_list.py`: `PiShowDuplicateGroup(rowget=self.get_selected_rows).item()`
- `pi_gallery_elem.py`: `rowget=self.get_rows`
- `pi_image_elem.py`: `rowget=self.get_row` (list of 0–1 row; anchor logic still works)

You can either:

- **Concatenate** after `get_show_submenu()`, e.g. `get_show_submenu() + [PiShowDuplicateGroup(...).item()]`, or  
- Extend `StatusMenu` with an optional “extra show items” parameter so you do not duplicate the concatenation in three files.

**Top menu bar** (`pi_menu.py` uses `StatusMenu()` with no `rowget`): if you add Duplicate Group there, you need a defined behavior (e.g. derive selection from `values[c.EVT_TREE]` when present, else status “select images in the tree first”).

## Summary

- Put **row-set logic** in `pi_filters.py` as a `Filter` that takes **precomputed `dup_target` / group** in `__init__`, analogous to `SelectedTreeNodesFilter` taking folder/file sets up front.
- Put **“read selection, build filter, call `filter_rows` + notify”** in a **new `PiAction` with per-instance event + `rowget`**, like `PiActionMap`, because global `PiFilterTable` + static `FilterTbd()` cannot see the current selection.
