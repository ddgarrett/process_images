## Cause: folder filter uses string prefix, not folder boundaries

The gallery builds its list with `SelectedTreeNodesFilter` in `pi_filters.py`. For each selected tree node that is a directory, it keeps rows whose `file_location` **starts with** that folder key:

```91:95:/Users/douglasgarrett/Documents/projects/process_images/pi_filters.py
    def test(self,row:Row):
        file_loc = row['file_location']
        for dir in self._filter_folders:
            if file_loc.startswith(dir):
                return True
```

That pattern is right for **nested** paths (e.g. selecting `/2018-11` should include `/2018-11/ipod touch/2018-11-01`). Tree folder keys are built as path segments joined with `/` (see `pi_treedata.py`).

It fails for **sibling** folders whose names only differ by extra characters after the same prefix. Example:

- Selected folder key: `/2024-11-03`
- Another folder: `/2024-11-03_am` or `/2024-11-03_copy`

Then:

- `'/2024-11-03_am'.startswith('/2024-11-03')` is **True** (the next character is `_`, not `/`, but `startswith` does not care).

So anything stored under a folder whose path **starts with the same characters** as `2024-11-03` will be included, even though it is not inside that directory on disk. Your suffix-qualified names are exactly that case.

So when you click the `2024-11-03` node, the filter intentionally treats “under this folder” as “`file_location` starts with this key,” which wrongly includes the longer sibling folder keys.

## Summary

The behavior comes from **`SelectedTreeNodesFilter.test()` using `file_loc.startswith(dir)`** without requiring a path separator after `dir` for subfolders (e.g. `dir == file_loc` or `file_loc.startswith(dir + '/')`). That is why one dated folder also pulls in rows whose `file_location` is another folder name that merely extends `2024-11-03` with a suffix.

### Update in `pi_filters.py`

`SelectedTreeNodesFilter` no longer treats every `file_location` that **string-prefixes** the selected folder as inside that folder. It now treats a row as in the folder when:

1. **`folder_key == ""`** (root) — still matches every row.
2. **`file_loc == folder_key`** — that folder itself.
3. **`file_loc.startswith(folder_key + "/")`** — a **child** path under that folder.

So `/2024-11-03_extra` no longer matches selection `/2024-11-03`, while `/2024-11-03/sub` still does.

Helper: `_row_in_folder()`; the loop variable was renamed from `dir` to `folder_key` to avoid shadowing the builtin.
