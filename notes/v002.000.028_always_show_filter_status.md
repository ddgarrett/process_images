The "squeezed toward the bottom" look is almost certainly **layout/Tk geometry**, not the text you pass to `update_status`. Your status row is set up like this:

```99:115:main.py
            sg.Text(
                "",
                relief=sg.RELIEF_SUNKEN,
                size=(55, 1),
                pad=(0, 3),
                key="-STATUS-",
                expand_x=True,
            ),
            sg.Text(
                "",
                relief=sg.RELIEF_SUNKEN,
                size=(28, 1),
                pad=(0, 3),
                key="-STATUS-FILTER-",
                justification="right",
            ),
            sg.Sizegrip(pad=(3, 3)),
```

A few things combine to push the glyphs down or clip them:

### 1. `size=(..., 1)` is only one text line tall
In PySimpleGUI/FreeSimpleGUI, `size` is **characters x lines**, not pixels. One line is often **tighter than the font + sunken border** need. The border takes vertical space inside the widget, so the label's baseline can sit low or look clipped at the bottom. That pattern was already in the app before the filter split; the examples use the same `size=(55, 1), pad=(0, 3)`.

### 2. `pad=(0, 3)` adds little vertical room
`pad` is `(left/right, top/bottom)` padding around the element. Three pixels top and bottom is minimal, so there is not much space above the text inside the sunken box.

### 3. One row, three widgets with different heights
The row has two `Text` elements plus a `Sizegrip`. Tk lays them out on one grid row; the row height tends to follow the **tallest** child (often the size grip). Shorter labels are then aligned within that taller row, which on many themes looks like text sitting on the **bottom** of the bar. Adding the second status `Text` can make that more obvious than a single full-width bar.

### 4. `justification="right"` only affects horizontal alignment
It does not vertically center text in the box.

Yes, that look is possible, even when padding is set symmetrically.

### Symmetric `pady` can still look top-heavy

Your code uses equal internal padding on both status labels:

```129:132:main.py
def _configure_status_label_padding():
    """Tk padx/pady: space between label text and the sunken border."""
    for key in ("-STATUS-", "-STATUS-FILTER-"):
        c.window[key].Widget.configure(padx=4, pady=2)
```

Tk treats `pady=2` as the same space above and below the text **inside the label**. Visually it can still feel like "extra room on top, tight on the bottom" because of other layout effects:

1. **Font metrics** -- A one-line label is sized for cap height and ascenders. Descenders (`g`, `y`, `p`) sit low and can sit close to or clip the bottom border, while there is often empty space above capital letters. That reads as "gap on top, cramped below" even with equal `pady`.

2. **Sunken border** -- `RELIEF_SUNKEN` draws a 3D inset frame. The bottom inner edge can look closer to the text than the top because of shading, not because `pady` is uneven.

3. **Row alignment with the size grip** -- The status row has two `Text` widgets plus a `Sizegrip`. The row height is often set by the tallest widget (usually the grip). The labels are then positioned in that taller row; on macOS/Tk that often anchors content toward the **bottom** of the cell, which makes any internal top space more obvious.

4. **Outer `pad=(0, 3)`** -- That adds space outside the widget in the grid. It does not balance text inside the sunken box and can add to the uneven feel.

5. **`size=(55, 1)`** -- One line of character height is a tight box. Small changes (font, theme `"black"`, HiDPI) easily make the bottom feel clipped while the top still has slack.

So: **yes, "too much top, not enough bottom" is a real and common artifact** here; it does not necessarily mean `pady` is wrong in code.

### What usually helps (if you want to tune in Agent mode)

| Approach | Effect |
|----------|--------|
| **Asymmetric internal padding** | e.g. `pady=0` via `configure`, then only add bottom: some use a trick with extra newline (bad for paths) or switch widget type |
| **Increase bottom only** | Tk `Label` does not support separate top/bottom `pady`; options are `pady` on the grid cell (`ipadx`/`ipady`), a `Frame` wrapper with padding, or `ttk.Label` / custom layout |
| **Slightly larger bottom `pady` experiment** | Try `pady=1` or `0` -- less top gap; or wrap label in `Frame` with `pack(pady=(0, 3))` for asymmetric inset |
| **`anchor='w'` + frame** | Put the label in a `Frame` with fixed height and `pack(anchor='center')` or `anchor='s'` to nudge text down visually |
| **Avoid `size` height 2** | You already saw that causes wrapping on long paths |

Practical next step if you want to fix the look: try **lowering or removing internal `pady`** (`pady=0` or `1`) first; if still top-heavy, use a **thin `Frame` around the label** with asymmetric `pack` padding `(top, bottom)` -- that is the reliable way to get "more room at bottom" when Tk's symmetric `pady` is not enough.

## What the layout is doing

The bottom row is three widgets in one grid row:

```98:116:main.py
        [
            sg.Text(
                "",
                relief=sg.RELIEF_SUNKEN,
                size=(55, 1),
                pad=(0, 3),
                key="-STATUS-",
                expand_x=True,
            ),
            sg.Text(
                "",
                relief=sg.RELIEF_SUNKEN,
                size=(38, 1),
                pad=(0, 3),
                key="-STATUS-FILTER-",
                justification="right",
            ),
            sg.Sizegrip(pad=(3, 3)),
        ],
```

In PySimpleGUI/FreeSimpleGUI, `size=(width, height)` is in **characters**, and it acts like a **minimum width** for that label.

So this row wants roughly:

- **55** character widths for `-STATUS-`
- **38** for `-STATUS-FILTER-`
- plus the size grip

That is a large minimum (often **700+ pixels** depending on font). When you make the window **narrower than that**, Tk does not shrink those minimums gracefully. The expandable left label (`expand_x=True`) is often the one that gets squeezed to **zero or near-zero width**, and the fixed-width pieces can clip or sit off-screen. It can look like the whole status line vanished.

That got worse when you split the bar: before, one `size=(55, 1)` bar shared the row with only the grip. Now you effectively need **55 + 38** character minimums in the same row.

Nothing in your resize handlers clears or hides the status widgets. `EVT_WIN_CONFIG` is buffered in `main.py` and used for gallery/image thumb resizing; it does not call `update_status("")` or similar.

## Why it feels intermittent

It often only happens when the window width drops **below that combined minimum**, or while dragging through sizes where Tk is fighting over column width. Wider again and the bar reappears.

Vertical shrink is less likely the cause: the main `Pane` has `expand_y=True`, but the status row is a separate row and usually keeps one line of height unless the window is extremely short.

## Directions to fix (Agent mode if you want it applied)

1. **Lower or remove character `size` minimums** on the status labels (especially `size=(55, 1)` on the left). Use a small initial size and rely on `expand_x=True` for the left bar only.
2. **Wrap the status row in a `Column`** with `expand_x=True` so the row participates in layout as one unit.
3. **Give the left status all elastic width** and keep the right filter at a modest fixed width (or cap it), e.g. left: `expand_x=True` without a huge `size`, right: smaller `size` or no `size`.
4. **Optional:** one combined right label is already bounded; long `Showing: To Be Determined (TBD)` may still need truncation or a smaller font on narrow windows.

Most direct fix for “disappears when resized narrow”: change `-STATUS-` from `size=(55, 1)` to something small (e.g. `(1, 1)` or `(10, 1)`) and keep `expand_x=True`, and optionally trim `-STATUS-FILTER-` `size` if it still steals too much width.
