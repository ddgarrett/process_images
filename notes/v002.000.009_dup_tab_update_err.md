## What actually throws

The failure is an unguarded lookup in `PiTreeData._update_rows`: every row in the change notification must already exist as a **file node** in the tree and in `PiTreeList._key_id_dict`. If not, you get `KeyError`.

```117:132:/Users/douglasgarrett/Documents/projects/process_images/pi_treedata.py
    def _update_rows(self,tree,key_id_dict,rows):
        ...
        for row in rows:
            parent = row['file_location']
            v      = row['file_name']
            ...
            key = f'{parent}/{v}'
            id = key_id_dict[key]
```

`key_id_dict` is only rebuilt when the tree is fully rebuilt on `EVT_TABLE_LOAD`:

```82:91:/Users/douglasgarrett/Documents/projects/process_images/pi_tree_list.py
    def update_list(self,event,values):
        ''' Update the tree data after a table is loaded or filtered '''
        self._tree_data = PiTreeData(c.table.rows())
        c.window[self.key].update(values=self._tree_data)
        self._key_id_dict = {v:k for k, v in self._tree.IdToKey.items()}

    def update_rows(self,event,values):
        ''' update display after row value changes '''
        rows = values[c.EVT_TABLE_ROW_CHG]
        self._tree_data._update_rows(self._tree,self._key_id_dict,rows)
```

So `_update_rows` assumes: “every row I am told to refresh is represented in the **current** tree.” That is not always true.

## Why your scenario produces a row that is not in the map

Two mechanisms work together:

### 1. Different menus use different row sources

- **Tree** right-click uses `PiTreeList.get_selected_rows`, which only maps **tree selection** into rows from `self._tree_data.rows` (the same list the tree was built from).
- **Gallery / Duplicates** right-click uses `get_rows` → `self._selected_rows` on that element.
- **Image** tab uses `get_row` → the single `_collection_row`, resolved with `get_row_for_fn`, which intentionally searches **`c.table._original_rows`**, not only the filtered rows used for the tree.

```7:21:/Users/douglasgarrett/Documents/projects/process_images/pi_util.py
def get_row_for_fn(filename):
    ...
    for row in c.table._original_rows:
        if row['file_name'] == name and row['file_location'] == loc:
            return row
```

So `EVT_TABLE_ROW_CHG` can list rows that **never had a tree node** under the current filter, or rows you are no longer supposed to show after a refilter.

### 2. Duplicates tab shows the whole cluster; the tree does not

The Duplicates tab builds its list from **`_original_rows`** and `DuplicateGroupFilter`, not from the filtered `c.table.rows()` that feed the tree:

```23:27:/Users/douglasgarrett/Documents/projects/process_images/pi_dup_elem.py
        selected_rows = self._get_selected_rows(event, values)
        all_rows = c.table._original_rows if c.table else []
        dup_target = resolve_duplicate_group_target(selected_rows, all_rows)
        if dup_target:
            self._collection_rows = DuplicateGroupFilter(dup_target).filter(all_rows)
```

So you routinely see **group members that are not file nodes in the current tree**. That by itself is fine until a status change tries to **patch the tree** for one of those rows via `_update_rows`.

### Why the KeyError names the **key** path (`PXL_20260119_...`)

`StatusMenuItem` applies the same status to **every** row returned by `rowget`:

```59:66:/Users/douglasgarrett/Documents/projects/process_images/status_menu_item.py
    def handle_event(self,event,values):
        rows = self._rowget(values)
        for row in rows:
            row['rvw_lvl'] = self._level
            row['img_status'] = self._status

        values[c.EVT_TABLE_ROW_CHG] = rows
        c.listeners.notify(c.EVT_TABLE_ROW_CHG,values)
```

So the **`rows` list can still include the key** even when you *think* you only meant the non-key, for example:

- **Multi-selected thumbs** on the Duplicates tab (the key thumb stays selected when you also select the non-key).
- Or **Image** tab / notification path still tied to the key file while the menu runs on another surface.

Then `update_rows` tries to update the **key** row again. That raises `KeyError` if, at that moment, the key has **no** entry in `_key_id_dict`.

The most typical way the **key** drops out of `_key_id_dict` after you marked it `duplicate` is:

- You had **Possible Good or Best** (`FilterPossibleGoodPlus`) active. A row with `img_status == 'dup'` and level `2` **no longer passes** that filter (see `pi_filters.FilterPossibleGoodPlus.test`).
- If anything triggers **`EVT_TABLE_LOAD`** (another **Show** filter, **Duplicate group** from the menu path that calls `filter_rows` + notify, reload collection, etc.), `update_list` rebuilds the tree from the **filtered** row list. The former key file is **no longer a tree node**.
- `gallery` / `dup` / `get_row_for_fn` can still hand you that **same row object** from `_original_rows` or from `_selected_rows`, so it appears in `EVT_TABLE_ROW_CHG` even though it is not in `key_id_dict`.

Even **without** a reload, a row that was **never** on the current filtered tree (but is in the duplicate group and in `_selected_rows`) would also cause the same kind of crash; your stack trace pointing at the **key** path fits “still in the `rows` list for the second menu action” plus “not in the current tree map”.

## Short summary

The bug is **not** “duplicate logic is wrong” per se; it is that **`_update_rows` assumes every updated row has a tree id**, while **status can be applied to rows resolved from `_original_rows` / multi-selected thumbs / image selection**, and **the tree only reflects the last filtered load**. When those diverge (especially after the key is `dup` under a Good-or-Best filter and the tree is rebuilt, or when the key is still in the batch of rows from the gallery), `key_id_dict[key]` raises `KeyError`.

I am in **Ask mode**, so I cannot patch it here; a robust fix is usually one or more of: skip rows missing from `key_id_dict`, trigger refilter + tree rebuild (and selection cleanup) when status changes under a filter, or unify row-getters so tree updates only run for rows that actually have nodes. If you want that implemented in-repo, switch to Agent mode.

**Cursor**


The smallest, lowest-risk fix is to **skip rows that are not in the tree** instead of indexing `key_id_dict` blindly.

In `PiTreeData._update_rows`, at the start of the loop (after you compute `key`), add:

```python
if key not in key_id_dict:
    continue
```

(or use `.get()` and only call `tree.Widget.set` when you have an `id`).

**Why this is the simplest:** one localized change, no new events, no refilter rules, no menu unification. Rows that have no node (off-filter, duplicate-only-in-gallery, etc.) simply do not get in-place tree cell updates; rows that **do** have nodes still update as today.

**Trade-off:** If `_key_id_dict` were ever wrong while the node still existed, you would silently skip an update. In normal flows it is rebuilt with the tree on `EVT_TABLE_LOAD`, so that is uncommon compared to the crash you are fixing.

Slightly broader (still simple): do the same check in `PiTreeList.update_rows` before calling `_update_rows`, or filter `rows` there. Putting it in `_update_rows` keeps all tree-update assumptions in one place.
