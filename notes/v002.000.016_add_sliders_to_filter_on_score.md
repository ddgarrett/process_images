### What adds a bit of care (not hard, but easy to get wrong)

1. **Keep two lists (or recompute cleanly)**

   - a cached list **after** tree/dup filtering but **before** MUSIQ filtering, then re-apply the threshold; or

4. **UX detail**  
   Whether moving the tree selection should reset the slider to 0 is a product choice; your spec (“default 0 = no filter”) only defines the initial state, not reset-on-navigation.

5. **Layout**  
   The bottom row already has page text and buttons (`pi_gallery_elem.py` around lines 101-108). You can add a **second** row with a centered slider, or squeeze into one row; either is minor layout work in FreeSimpleGUI.

### Rough scope

- **Touch points**: mainly `pi_gallery_elem.py` (layout, listener, filter helper, maybe small hook from `files_selected`); possibly a tiny helper module or reuse of `parse_musiq_score` import. Duplicates likely needs **no** separate layout if it stays a subclass.

Overall: **not difficult** for someone who already knows this codebase; the work is localized, and the listener/slider pattern already exists for PgUp/PgDn-style keys. The only “design” piece is caching pre-filter rows and defining behavior for missing scores.

**Cursor**

You can do it **only if** the strings you compare are normalized so that **lexicographic order matches numeric order**.

### When plain string compare works

If every non-blank `musiq_score` is exactly the pattern you described (one digit before the dot and three after, e.g. `6.250`, `0.000`), and scores never go to `10.000` or higher, then:

- Padding the slider to the same pattern (e.g. threshold `6` → `"6.000"`) and using **`>`** for “strictly above” is consistent with numeric order for those strings.

### When it breaks

- **Two-digit whole part**: `"10.000"` vs `"6.000"` as strings gives `"10..." < "6..."` because `'1' < '6'`, which is wrong numerically. So you either assume scores stay in `0.000`–`9.999`, or you **zero-pad** the integer part to a fixed width on both sides (e.g. `06.500` vs `10.000` with 2-digit integer part) before comparing.

- **Format drift**: If anything ever appears as `6.5`, `6.50`, `06.250`, or scientific notation, string rules stop matching float rules unless you normalize first.

### Practical takeaway

Yes, you *can* avoid float parsing **if** you treat “same format” as a contract and optionally **normalize** (same decimal places + fixed-width whole part) before comparing. If that contract is not guaranteed forever, numeric parse (or normalize-then-compare) stays safer.

Your project already documents MUSIQ roughly in a 1–9.999 style in metadata, but the loader and tree treat `musiq_score` as **strings** from CSV in several places, so a string-based filter is plausible **if** you enforce one canonical format at import or when filtering.

## Cause

When you pick an image on the **Gallery** tab by clicking a thumbnail, the gallery notifies listeners with a **tiny** `values` dict that only contains the image select event, not the full window state:

```217:222:pi_gallery_elem.py

            filename = get_fn_for_row(self._selected_rows[-1])
            values = {c.EVT_IMG_SELECT:[filename]}
            c.listeners.notify(c.EVT_IMG_SELECT,values)
```

**Duplicates** subscribes to that same event:

```48:49:main.py
    gallery = PiGalleryElem(key="-GALLERY-", events=[c.EVT_TREE])
    dup = PiDupElem(key="-DUP-", events=[c.EVT_TREE, c.EVT_IMG_SELECT])
```

So `PiDupElem.files_selected` runs with that minimal `values`. Then `_apply_musiq_filter(values)` calls `_musiq_threshold(values)`, which does:

```229:233:pi_gallery_elem.py
    def _musiq_threshold(self, values):
        try:
            return float(values.get(self._musiq_slider_key, MUSIQ_SLIDER_MIN_NO_FILTER))
        except (TypeError, ValueError):
            return MUSIQ_SLIDER_MIN_NO_FILTER
```

The Duplicates slider key (e.g. `-DUP-MUSIQ-`) is **not** in that dict, so the code always falls back to **`MUSIQ_SLIDER_MIN_NO_FILTER` (3.0)**, which means **no MUSIQ filter** for that rebuild. Opening the Duplicates tab afterward does not run `files_selected` again, so you keep that unfiltered list.

So the bug is not “tab order” by itself; it is **rebuilding the dup gallery from `EVT_IMG_SELECT` while `values` has no slider entry**.

## Fix direction (conceptual)

Whenever you need full widget state (slider, tab group, tree, etc.) inside a listener that may get a **partial** `values` dict, merge with the last full snapshot your app already keeps on each `window.read()`:

- `main.py` assigns **`c.last_window_values = values`** when `values` is not `None` (see around line 151 in `main.py` and `pi_config.last_window_values`).

A typical approach is:

- In **`_musiq_threshold`** (or at the start of **`_apply_musiq_filter`** / **`files_selected`** for elements that read the slider), build something like **`effective = {**(c.last_window_values or {}), **values}**` (or the reverse order, depending on whether the partial dict should override keys), then read **`effective.get(self._musiq_slider_key, ...)`**.

Alternatively, change the gallery’s **`EVT_IMG_SELECT`** notify to pass **`{**c.last_window_values, **{EVT_IMG_SELECT: [...]}}`** so every listener still sees the full window values plus the new selection.

Either way, the Duplicates slider value must be present when `_apply_musiq_filter` runs after a gallery thumbnail selection.
