## Why it is straightforward

The toggle handler already shows exactly how to update both state and UI:

```191:198:pi_tree_list.py
    def _toggle_tree_score_comparator(self, event, values):
        self._score_cmp_less_than = not self._score_cmp_less_than
        c.tree_score_cmp_less_than = self._score_cmp_less_than
```

So `_show_filter_selected` would add the same pattern, but force `False` instead of toggling:

1. `self._score_cmp_less_than = False`
2. `c.tree_score_cmp_less_than = False` (gallery uses this via `get_tree_visible_rows`)
3. Update button text to `"Score >="` and blue button color

No new events, keys, or UI elements.

## One small wrinkle: callback order

On a Show filter event, listeners run in registration order. `PiFilterTable` registers at import time and, inside its handler, fires `EVT_TABLE_LOAD`, which rebuilds the tree **before** `_show_filter_selected` runs.

That means `update_list` may rebuild once while the comparator is still `Score <`. Fix: after resetting the comparator in `_show_filter_selected`, call the same refresh path used elsewhere (`_rebuild_tree`, `_update_status_summary`, and probably `_notify_tree_selection_with_threshold` with threshold `3.0`). That is a few lines copied from `_tree_score_changed` / `_toggle_tree_score_comparator`.

## Scope matches your request

This only runs for Show filter events (`EVT_SHOW_ALL`, `EVT_SHOW_TBD`, etc.), not when you manually drag the slider to 3.0. That matches "when the slider is set to 3.0 **by a new filter being selected**."

## Why it matters at 3.0

At 3.0, the two modes behave very differently:

- `Score >=` at 3.0: effectively "show scored images" (min score)
- `Score <` at 3.0: only images strictly below 3.0

If you pick a new Show filter while still in `Score <`, the slider resets visually but filtering stays in `<` mode until you toggle the button again.

## Optional cleanup (not required)

You could extract a small helper like `_set_score_comparator(less_than, values)` shared by `_toggle_tree_score_comparator` and `_show_filter_selected` to avoid duplicating the button update block. That is nice-to-have, not necessary for the feature.

**Bottom line:** low-risk, localized change -- extend `_show_filter_selected` with comparator reset + one extra rebuild so the tree and gallery pick up `Score >=` immediately after a filter change.

Yes. The double rebuild only happens if you reset the comparator in `_show_filter_selected` and then call `_rebuild_tree` again. Today there is actually **one** rebuild already -- the problem is **timing**, not too many rebuild calls.

## Current order

On a Show filter event:

1. `PiFilterTable.handle_event` runs first (registered at import in `pi_config.py`)
2. Inside that, `EVT_TABLE_LOAD` fires and `update_list` **rebuilds once**
3. `_show_filter_selected` runs after that (slider/button widget updates only)

So if you add comparator reset + another `_rebuild_tree` in step 3, you get two rebuilds.

## Ways to keep a single rebuild

### 1. Reset state before `update_list` runs (best fit)

Make sure comparator + threshold are correct **before** the existing `update_list` rebuild, not after.

**Option A -- flag from `PiFilterTable`:**

```python
# PiFilterTable.handle_event, before EVT_TABLE_LOAD
c.show_filter_score_reset = True
c.table.filter_rows(self._filter)
c.listeners.notify(c.EVT_TABLE_LOAD, values)
```

```python
# update_list
if c.show_filter_score_reset:
    c.show_filter_score_reset = False
    self._score_cmp_less_than = False
    c.tree_score_cmp_less_than = False
    threshold = MUSIQ_SLIDER_MIN_NO_FILTER
    # update slider + button widgets, sync tab sliders
    merged = {..., self._score_slider_key: threshold}
    self._rebuild_tree(merged)
    ...
else:
    # existing path
```

Then `_show_filter_selected` can be removed or left empty -- all logic lives in one rebuild path.

**Option B -- run `_show_filter_selected` first:**

Add `prepend` to `EventListener.add` and register the show handler at the front. In `_show_filter_selected`, reset comparator state and widgets, but **do not rebuild**. Then `PiFilterTable` -> `update_list` rebuilds once.

You still need to force threshold `3.0` into the merged values used by `update_list`, because widget updates alone do not change `values` / `last_window_values`. A small flag like `c.force_tree_threshold = 3.0` that `update_list` reads and clears works.

### 2. Do not rebuild in `_show_filter_selected`

If you only set state + button in `_show_filter_selected` without `_rebuild_tree`, you still have one rebuild -- but it already happened in step 2 with the **old** comparator/threshold. So the tree stays wrong until something else triggers a rebuild. That avoids double rebuild but does not fix the bug.

## Recommendation

Use **one coordinated path through `update_list`**:

- Set a flag when a Show filter is applied (`PiFilterTable` is the natural place).
- In `update_list`, when the flag is set: reset to `Score >=`, threshold `3.0`, update widgets, rebuild once.
- Drop duplicate work from `_show_filter_selected`.

That gives you one rebuild and correct filtering immediately.

## What not to do

Avoid "rebuild in `update_list`, then rebuild again in `_show_filter_selected`" -- that is the redundant path.

Also avoid only updating the button in `_show_filter_selected` and hoping the earlier rebuild was correct; it was not, because comparator state was still stale when `update_list` ran.

## Suggested helpers

Two focused helpers is the sweet spot:

```python
def _set_score_comparator(self, less_than: bool) -> None:
    self._score_cmp_less_than = less_than
    c.tree_score_cmp_less_than = less_than
    if c.window is not None and self._score_cmp_btn_key in c.window.AllKeysDict:
        c.window[self._score_cmp_btn_key].update(
            text=self._score_cmp_button_text(),
            button_color=self._score_cmp_button_color(),
        )

def _refresh_tree_for_threshold(self, values, threshold: float) -> None:
    base = c.last_window_values if c.last_window_values is not None else {}
    overlay = values if values is not None else {}
    merged = {**base, **overlay}
    merged[self._score_slider_key] = threshold
    self._rebuild_tree(merged)
    self._update_status_summary(merged)
    self._notify_tree_selection_with_threshold(merged, threshold)
```

Then the handlers shrink to:

```python
# update_list (show-filter branch)
self._set_score_comparator(False)
threshold = MUSIQ_SLIDER_MIN_NO_FILTER
# update tree slider widget only here
self._sync_tab_sliders(threshold)
self._refresh_tree_for_threshold(values, threshold)

# _toggle_tree_score_comparator
self._set_score_comparator(not self._score_cmp_less_than)
threshold = self._tree_slider_value(values)
self._sync_tab_sliders(MUSIQ_SLIDER_MIN_NO_FILTER)
self._refresh_tree_for_threshold(values, threshold)
```

`_tree_score_changed` could use `_refresh_tree_for_threshold` too, since it ends with the same rebuild trio.

## What not to combine

Avoid one big helper like `_apply_score_control_change(reset_comparator=..., reset_tree_slider=..., sync_tabs=...)` -- the two callers differ enough that a parameter-heavy function would be harder to read than two small ones.

Also keep the **tree slider widget update** only in the show-filter path; toggle intentionally does not move the tree slider.

## Net effect

- Handlers read as "configure state, then refresh"
- Comparator UI logic lives in one place
- Rebuild/notify logic lives in one place
- No new globals or cross-module flags

So yes: a helper (really two) is a good fit here and would make the structure slightly simpler locally, without undoing the single-rebuild design you already have.
