**Cursor**


Yes. The only reason is **when** tkinter gets loaded in the **main** process.

- **`import FreeSimpleGUI`** (and **`import pi_config`**, since it imports FreeSimpleGUI) pulls in **tkinter** in the same interpreter that‚Äôs running `main.py`.
- On some macOS setups, loading tkinter in that process can **abort** (e.g. old/broken Tcl) before your code can print a helpful message.

So anything that **transitively imports tkinter** must run **only after** you know Tcl/Tk is OK.

The subprocess in `_check_tcl_version` is deliberate: it runs `import tkinter` in a **child** process. If that child dies, the parent is still alive and can `print(...)` and `sys.exit(1)`.

**`tcl_check` is safe to import first** because it only uses `sys` and `subprocess` ‚Äî no tkinter in the main process until after `_require_tcl_86()` returns.

So: there *is* a reason for ‚ÄúGUI-related imports after the check,‚Äù but it‚Äôs satisfied by **importing `tcl_check` first** (check on import) and then the rest at the top, not by scattering imports mid-file anymore.
