## Why Linux (and sometimes Mac) feels “stuck” (q, then y)

On Unix, CPython’s `webbrowser` module registers several CLI tools as normal `UnixBrowser` instances, including **w3m**, **lynx**, **links**, and **elinks**. For that class, `open()` uses `os.spawnvp(..., P_WAIT, ...)` **unless** it is a `BackgroundBrowser`. So Python **waits until that process exits**.

Flow:

1. No usable GUI browser is chosen first (or your `BROWSER` env points at a TUI browser).
2. Python falls through to something like **w3m** / **links** / **lynx**.
3. That opens **in the terminal**, holds the foreground, and **blocks** your app until you quit.
4. In **w3m** (and similar), **q** starts exit and you often get a **yes/no** prompt, hence **y** - that matches what you are seeing.

On **Windows**, the default path usually goes through **`os.startfile`**-style handling, which does not do this “wait for w3m” dance, so you do not see the hang.

## Why the “map” does not show

The HTML from **gmplot** is a **Google Maps / JavaScript** page. Terminal browsers **do not run that JS** (or only in a very limited way), so you get a broken or empty page even though a “browser” did open.

So you get two symptoms from the same root cause: **wrong handler** (TUI browser) + **blocking wait** in `webbrowser`.

## Mac-specific nuance

On macOS, `webbrowser` normally prefers **AppleScript** (`open location ...`) and usually returns quickly. If something overrides that (environment, broken default, odd PATH), you could still hit a similar fallback. Separately, passing a bare filesystem path instead of a proper **`file://...`** URL can make some handlers behave badly on Unix; Windows is more forgiving of a plain path for `file` open.

## Linux: why it hangs and why there is no map

Python’s `webbrowser` module tries a **fallback list** of programs on Unix. When no preferred GUI browser wins early, it can pick **lynx**, **w3m**, **links**, etc. Those are registered as **`UnixBrowser`**, which launches the executable with **`os.P_WAIT`** (wait until the process exits). So your main program **blocks until you quit Lynx** with `q` (and any follow-up prompt).

Lynx **cannot run Google Maps / gmplot’s JavaScript**. At best you see the HTML structure or a useless page, not an interactive map. So the file is fine; the **handler** is wrong.

Double-clicking `temp_map.html` uses your **desktop default for `.html` files** (Chrome, Firefox, etc.), which is why that path works.

## Mac: why there is no hang but still no map in-app

On macOS, `webbrowser` normally goes through **`osascript` / `open location`**, which **returns quickly**, so you do not get the Lynx-style blocking behavior.

If the map still does not show when triggered from code but **does** show when you open the same file from Finder:

- **Different open path:** `webbrowser.open(...)` may not pass the same **`file:///...`** URL that Finder uses, so the GUI browser can open an empty tab, the wrong document, or treat the path oddly.
- **Different app:** Code might open your **HTTP(S) default browser** (or whatever AppleScript ties to `open location`) while double-click uses **default app for `.html`**, which can differ.
- **Less common:** If the tab does load `file://...` but the map is blank, some browsers are stricter about **local files + API keys / mixed content** than others; that would explain “works in one browser from Finder, blank when opened another way.”

So on Mac the story is usually **“wrong URL shape or wrong handler,”** not “HTML not written.”

## Takeaway

| OS   | What you observe | Likely cause |

| Linux | Lynx UI in the terminal + hang until `q` | `webbrowser` chose Lynx (or similar), **blocks** with `P_WAIT`, cannot render JS maps. |
| Mac   | No hang, no map in-app; double-click OK | `webbrowser` vs Finder: **URL / default app** mismatch, not a bad `temp_map.html`. |

Here is what changed.

### New helper: `_open_local_html`

- **macOS:** runs `open <absolute-path>` in a detached `Popen` (same idea as double-click in Finder), with stdout/stderr discarded so the terminal stays clean.
- **Windows:** uses `os.startfile()` on the resolved path (same as before in practice, without going through `webbrowser`).
- **Linux and other Unix:** uses `xdg-open` when it is on `PATH`, also detached and non-blocking.
- **Fallback** if there is no `xdg-open:` `webbrowser.open_new_tab(path.as_uri())` so a proper `file://` URL is used instead of a bare path.

### Call site

`webbrowser.open_new_tab(map_fn)` was replaced with `_open_local_html(map_fn)` after `gmap1.draw(map_fn)`.

`webbrowser` stays imported for that fallback only.

You should see the map open in your normal GUI browser on Mac and Linux, with no Lynx session and no hang on Linux. If you ever run on a minimal Unix image without `xdg-open`, install `xdg-utils` or rely on the `file://` fallback (still better than a raw path, but can still pick a text browser if nothing else is configured).
