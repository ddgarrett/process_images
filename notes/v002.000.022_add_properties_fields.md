## What your CSV shows

1. **`2024-11_NYC_prelim_map.jpg` (row 1000)**

So the question is less “does this camera store WxH somewhere odd?” and more “**why are `Image ImageWidth` / `EXIF ExifImageWidth` (and the length pair) missing for these JPEGs while other tags remain?**”

## How your app fills width/height

From `image_collection_metadata.csv`, width and length come only from:

```8:9:/Users/douglasgarrett/Documents/projects/process_images/image_collection_metadata.csv
img_width,int,0,get_key,"[""Image ImageWidth"",""EXIF ExifImageWidth""]",
img_len,int,0,get_key,"[""Image ImageLength"",""EXIF ExifImageLength""]",
```

`exif_loader._get_key` returns the **first matching tag** in that list, else the default `0`:

```260:267:/Users/douglasgarrett/Documents/projects/process_images/exif_loader.py
    def _get_key(self, tags: dict[str, any], exif_tags: any, default: any):
        if type(exif_tags) == str:
            return tags.get(exif_tags, default)

        for tag in exif_tags:
            if tag in tags:
                return tags[tag]

        return default
```

So **if those tags are absent** (stripped, never written, or only present under another EXIF name), you keep **0** even when the file is a normal JPEG.

## Do cameras store dimensions “somewhere else”?

**In the file, yes: pixel dimensions always exist in the JPEG itself** (Start Of Frame / SOF markers). That does not depend on EXIF. Your pipeline simply **never reads SOF**; it only reads selected EXIF tags.

**In EXIF**, width/height are *usually* in IFD0 (`Image ImageWidth` / `Image ImageLength`) and/or the EXIF sub-IFD (`EXIF ExifImageWidth` / `EXIF ExifImageLength`). Many tools also expose **`EXIF PixelXDimension` / `EXIF PixelYDimension`** (same semantic role). Your metadata list does **not** include those names, so if `exifread` only surfaces PixelXDimension/PixelYDimension for some marathon files, you would get **0** while Make/Model still fill from other tags.

Vendor **MakerNote** sometimes has extra image-size-related fields, but for typical JPEGs the **main** gap is either missing IFD0/EXIF dimension tags in the delivered file, or **different tag names** than the ones you query—not that Nikon/Canon only store size in a secret MakerNote block for normal photos.

## Practical takeaway for the marathon-style rows

Those rows look like **vendor-delivered or re-encoded JPEGs** that keep **camera/date EXIF** but **omit or rename** the dimension tags your loader looks for. That is a **file/pipeline** issue, not a brand quirk.

**Options** (conceptually): extend the EXIF tag list (e.g. try `EXIF PixelXDimension` / `EXIF PixelYDimension`), and/or **fallback to decoding dimensions** with an imaging library (e.g. open the file and read `.size` from the raster) when EXIF width/length are still 0.

So: **yes, dimensions can still “be there” in the file** (JPEG structure); **they might also appear under other EXIF tag names** your CSV does not list yet; it is **unlikely** that the camera model alone explains 0s without also checking what tags those actual files contain (e.g. with `exifread` or `exiftool` on one sample from `marathon_purchased`).

---

## Making it stay above the main window

### 1. `keep_on_top=True` (easiest)

**Difficulty: very low** - add one keyword to `sg.Window`.

Behavior: the Properties window is marked **system topmost**, so it tends to stay above **everything**, including other apps, until it closes. That is stronger than “only above the main window,” but it is the straightforward built-in knob and matches how your demos use `keep_on_top` (e.g. `tests/psg_demo_all_elements.py`).

Tradeoff: can feel heavy-handed (always on top of *all* windows).

### 2. Tie the dialog to the main window (Tk `transient`)

**Difficulty: low to moderate** - usually a few lines **after** `finalize=True`, using the underlying Tk widgets.

FreeSimpleGUI keeps `c.window` as the main app window (`main.py` assigns `c.window = init_window()`). After creating and finalizing the Properties window, you can call Tk APIs on `window.TKroot` with parent `c.window.TKroot` (e.g. **`transient(parent)`**). That ties the dialog to the main window for stacking/window-manager behavior; it is **not** the same as global topmost, but it often improves “stays with the app” behavior.

Exact behavior varies by platform (macOS vs Windows).

### 3. Validation popups

Your handler uses **`sg.popup(err)`** for GPS validation errors. If you add **`keep_on_top=True`** only on Properties, those popups might still appear **behind** the Properties window unless you also pass **`keep_on_top=True`** (or similar) to those popups. Worth testing after any change.

## Summary

| Goal | Effort |
|------|--------|
| Strong “always on top” while open | **Trivial**: `keep_on_top=True` on the Properties `sg.Window` |
| Only “above main,” less global | **Small Tk snippet**: `transient` on `TKroot` (and test on macOS) |
| Modal semantics | **Already there** with `modal=True`; stacking is a separate concern |

So: **not difficult** - often one parameter for the common case; a bit more care if you want strict “above main only” and consistent popups.
