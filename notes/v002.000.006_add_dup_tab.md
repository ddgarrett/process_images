```python
def _status_if_active(self, values, tab_key, msg):
    """Update status only when the requested tab is active."""
    current_tab = values.get("-TABGROUP-") if isinstance(values, dict) else None
    if current_tab:
        if current_tab == tab_key:
            c.update_status(msg)
        return

    # Fallback when event values do not include -TABGROUP-
    if c.window is not None and "-TABGROUP-" in c.window.AllKeysDict:
        try:
            if c.window["-TABGROUP-"].get() == tab_key:
                c.update_status(msg)
        except Exception:
            pass
```



Why this shape:
- Checks `values["-TABGROUP-"]` first (cheap and event-local).
- Falls back to reading current tab from the window when `-TABGROUP-` is missing from event values.
- Fails safe (no status update if tab cannot be determined).

## What `dup_target` means in data

If **any other row** has `dup_photo` pointing at this row (same normalized path as `row_dup_ref(this_row)`), then this row is the **canonical side** of at least one duplicate link. Setting `dup_target` to `"T"` in that case matches your description and matches the second branch of today’s `resolve_duplicate_group_target` (the “referenced by another row’s `dup_photo`” case).

## Setting it in `exif_loader`

Right after `apply_musiq_scores_csv` has written `dup_photo` on each matched row (and you have a full set of rows in `table`), you can:

1. Scan all rows and collect normalized keys from every non-blank `dup_photo`:
   - `targets = { normalize_dup_path_key(r.get("dup_photo")) for r in rows if str(r.get("dup_photo") or "").strip() }`
2. For each row `r`, set  
   - `r["dup_target"] = "T"` if `normalize_dup_path_key(row_dup_ref(r)) in targets`  
   - else set to your default (likely `""` to match metadata default `F` in the CSV description, or literal `"F"` if you want explicit).

You only need this when the scores file path runs (same guard as today: file exists, required columns, etc.). For rows not present in the scores CSV, you can leave `dup_target` as default, or clear it, depending on whether `dup_photo` can only come from that file in your workflow.

**Reuse:** Implement step 1–2 once in `pi_dup_group.py` (e.g. `refresh_dup_target_column(rows)` or `apply_dup_target_flags(table)`) and call it from `apply_musiq_scores_csv` at the end. That reuses `row_dup_ref` and `normalize_dup_path_key` and keeps one definition of “who is referenced as a target.”

## Can `dup_target` simplify `pi_dup_group`?

**Yes, for `resolve_duplicate_group_target` only.**  
`DuplicateGroupFilter` should stay as it is: it still must include every row where `dup_photo` matches the group key or the row is the canonical path. The `"T"` flag does not replace that logic.

Simplified `resolve_duplicate_group_target`:

- For each selected row in order (same as now):
  - If `dup_photo` is non-blank → return `normalize_dup_path_key(dp)` (unchanged).
  - Else if `dup_target == "T"` (or whatever sentinel you use) → return `normalize_dup_path_key(row_dup_ref(row))` **instead of** scanning all rows with `any(...)`.

That removes the O(selected x all_rows) “is anyone’s dup_photo equal to my ref?” scan **when `dup_target` is up to date**.

## Caveats

1. **Freshness:** If the user edits `dup_photo` in the app (or CSV) without recomputing `dup_target`, the fast path can be wrong. Mitigation: re-run the same `apply_dup_target_flags` after saves or when merging scores data.

So: populate `dup_target` in `exif_loader` using shared helpers from `pi_dup_group`, and optionally simplify `resolve_duplicate_group_target` to prefer the column when present, with a documented recompute policy so the column stays trustworthy.
