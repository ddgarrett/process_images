## Row data / display rules

`cosine_sim` is a normal column on collection rows (see metadata / import). Your rule matches how the field is documented: treat **blank** (unknown) and **`-1`** (non-duplicate) as hide; otherwise show the **string as stored** (no conversion). You can optionally also require `img_status == "dup"` or non-empty `dup_photo` if you want to restrict to "duplicate of another" beyond the cosine cell alone; for typical data, "not blank and not `-1`" already implies a stored duplicate similarity.

Why do we need the strip_text:bool?

---

**Cursor**


You do **not** need a `strip_text` flag unless you fold **both** behaviors into **one** draw helper and want to keep them different.

**Why two behaviors exist**

- **Left overlay** (`_draw_overlay_text`) calls `text.strip()` before measuring and drawing.
- **Right overlay** (`_draw_overlay_text_right`) uses `text` as given so `cosine_sim` stays **as stored** (your earlier requirement).

A single internal function with `strip_text=True/False` is one way to say: “left path normalize whitespace, right path do not.”

**When you can drop `strip_text`**

- If the shared helper **never** strips and **left** text is already normalized at the source: `get_overlay_text()` already ends with `.strip()` on the combined string, so for the only current left call site the extra strip inside `_draw_overlay_text` is mostly redundant. Then unified code could avoid stripping entirely, and you still would not need a flag.
- If the shared helper **always** strips, you lose “as-is” cosine whenever the stored value has leading/trailing spaces you care about (usually you do not).

So **`strip_text` is not a requirement of the problem**; it was only a way to merge two functions without changing either behavior. A merge without that flag is fine if you pick one rule (no strip in the drawer + callers responsible, or strip everywhere and accept cosine display matching that rule).
