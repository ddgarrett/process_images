## Event path

## What “selection” means

In `files_selected`, the gallery reads **the current tree multi-selection** from the event key (same as `-TREE-`):

```155:172:/Users/douglasgarrett/Documents/projects/process_images/pi_gallery_elem.py
    def files_selected(self,event,values):
        ''' Display a new list of collection rows
         
            Get the list of rows from current table, 
            BUT - in future may have to make a change to get 
            a filtered list of rows instead?
        '''

        self._base_collection_rows = []
        self._collection_rows = []
        self._selected_rows = []
        self._page = 0

        files_folders = values[event] # files and folders to display
        rows = c.table.rows()
        if len(rows) > 0 and len(files_folders) > 0:
            filter = SelectedTreeNodesFilter(files_folders)
            self._base_collection_rows = filter.filter(rows)
```

So the gallery does **not** scan disk when you pick a folder. It takes the list of **tree node keys** (paths built in `PiTreeData`: folder nodes like `/2018-11/...`, file nodes like `/path/to/folder/IMG.jpg`) and filters **`c.table.rows()`** with `SelectedTreeNodesFilter`.

## How a folder maps to rows

`SelectedTreeNodesFilter` splits each selected key into folder vs file:

```76:97:/Users/douglasgarrett/Documents/projects/process_images/pi_filters.py
        for name in selected:

    def test(self,row:Row):

                return True
            
        file_name = row['file_name']

```

- If the key refers to a **directory** under `c.directory`, it is treated as a **folder**: every table row whose `file_location` **starts with** that folder key is included (the root `""` selection is special-cased to match via `startswith` as well).
- Otherwise the key is treated as a **file** node: the row must match `(file_location, file_name)` after splitting the path.

Tree keys and `file_location` come from the same scheme (`PiTreeData` builds folder parents from each row’s `file_location` and inserts files under that parent).

## Extra step: MusIQ slider

After that set is chosen, `_apply_musiq_filter` keeps only rows whose `musiq_score` is strictly above the gallery MusIQ slider threshold before thumbnails are drawn:

```242:247:/Users/douglasgarrett/Documents/projects/process_images/pi_gallery_elem.py
    def _apply_musiq_filter(self, values):
        thr = self._musiq_threshold(values)
        self._collection_rows = []
        for row in self._base_collection_rows:
            if _musiq_cell_strictly_above_threshold(row.get("musiq_score"), thr):
                self._collection_rows.append(row)
```

**Note:** `GalleryTab` in `pi_gallery_tab.py` is a separate stub (`_update_gallery` only prints). The behavior above is entirely in `PiGalleryElem` plus `SelectedTreeNodesFilter`.

Tracing how the tree is populated versus how the gallery filters rows so we can compare them.



Here is a concise comparison.

## Same base: `Show` filter and table rows

Both the tree and the gallery start from **`c.table.rows()`**, which is the table after the **Show** menu filter (All, TBD, Possible Duplicate, etc.). So status/review filtering is shared; neither side shows rows that are excluded by that filter.

## Where they diverge

### 1. What appears **under a folder in the tree**

The tree is built from **`_visible_rows`**, not from raw `c.table.rows()` alone:

```275:277:/Users/douglasgarrett/Documents/projects/process_images/pi_tree_list.py
    def _rebuild_tree(self, values):
        selected_keys, expanded_keys = self._snapshot_tree_state()
        self._tree_data = PiTreeData(self._visible_rows(values))
```

`_visible_rows` takes `c.table.rows()` and then keeps only rows whose **`musiq_score`** passes the **tree** controls: the **Tree Score** slider (`-TREE_SCORE-`) and the **Tree Score &lt; / &gt;=** toggle (strictly below vs at or above the threshold):

```220:231:/Users/douglasgarrett/Documents/projects/process_images/pi_tree_list.py
    def _visible_rows(self, values):
        rows = c.table.rows() if c.table else []
        thr = self._tree_slider_value(values)
        visible = []
        for row in rows:
            if self._score_cmp_less_than:
                keep = _musiq_cell_strictly_below_threshold(row.get("musiq_score"), thr)
            else:
                keep = _musiq_cell_at_or_above_threshold(row.get("musiq_score"), thr)
            if keep:
                visible.append(row)
        return visible
```

So the **files listed under a folder** are only rows that:

1. Pass the current **Show** filter, and  
2. Pass the **tree** MusIQ rule (and only those rows are passed into `PiTreeData`, so folder nodes only get file children from this set).

### 2. What the **gallery** shows for a selected folder

The gallery still uses **`c.table.rows()`** (same Show filter), then **`SelectedTreeNodesFilter`** for the selected tree keys (folder prefix / file match). It does **not** re-apply the tree’s `_visible_rows` step. After that it applies the **gallery** MusIQ slider (`-GALLERY-MUSIQ-`) with **`_musiq_cell_strictly_above_threshold`** (score **strictly greater than** the gallery threshold), not the tree slider or the tree &lt;/&gt;= toggle.

So gallery thumbnails are:

1. Same **Show** filter as the table.  
2. Rows under the selection by path (same idea as folder membership in the data model).  
3. Filtered by the **gallery** score control and **strict &gt;** semantics, independent of whether the tree hid that row.

### 3. Practical differences

| Aspect | Files under folder in tree | Gallery for selected folder |
|--------|----------------------------|-----------------------------|
| Show filter | Yes (`c.table.rows()`) | Yes (`c.table.rows()`) |
| MusIQ source | Tree slider + &lt;/&gt;= mode | Gallery slider only |
| MusIQ rule | At or above **or** strictly below (tree mode) | Strictly **above** threshold |
| Hidden files | Rows failing tree score never get file nodes | Rows failing gallery score still omitted from thumbnails |

So the tree can **omit** a file under a folder (tree score filter) while the gallery could still show it if its slider is looser (and vice versa). Sliders are sometimes synced from the tree (`_sync_tab_sliders` on show-filter changes and related paths), but they are separate controls and the **&lt; vs &gt;=** behavior exists only on the tree side.

**Bottom line:** Under-folder **listing** is “Show filter + tree MusIQ visibility.” The **gallery** is “Show filter + folder selection + gallery MusIQ,” without applying the tree’s `_visible_rows` pass.

what would it take to make it so that the gallery only shows the same files that would be shown under a folder in the treeview, before the gallery slider is applied?

---

**Cursor**

Here is a concrete picture of what would need to change.

## Goal

Today the gallery builds `_base_collection_rows` from **`c.table.rows()`** plus **`SelectedTreeNodesFilter`**. The tree only builds file nodes from **`_visible_rows(values)`** (same table rows, then tree MusIQ rules). You want **`_base_collection_rows`** to match “what would appear under that folder in the tree” **before** `_apply_musiq_filter` (gallery slider).

That means: **`_base_collection_rows` should be `SelectedTreeNodesFilter(selection).filter(visible_rows)`** where **`visible_rows`** uses the **same** logic as **`PiTreeList._visible_rows`**.

## What you need

### 1. One shared definition of “tree-visible” rows

`_visible_rows` lives on `PiTreeList` and applies:

- `c.table.rows()` (already respects Show filter)
- Tree score threshold from **`values`** (`-TREE_SCORE-` merged with `c.last_window_values` in `_tree_slider_value`)
- Comparator mode: **`_score_cmp_less_than`** vs default (`>=`)

So anything that replaces `c.table.rows()` in the gallery path must use **that** logic, not a copy-pasted variant that can drift.

Reasonable approaches:

- **Extract** a module-level function (e.g. `visible_rows_for_tree_score(rows, values, cmp_less_than)`) used by both `PiTreeList._visible_rows` and the gallery, or  
- **Inject** the tree instance into the gallery (e.g. from `_ui_refs` in `main`) and call **`tree._visible_rows(values)`** from `files_selected`, then run `SelectedTreeNodesFilter` on that list.

Without one of these, you will not reliably match the tree.

### 2. Access to the tree comparator flag

`_score_cmp_less_than` is **instance state** on `PiTreeList`, not in `values`. The gallery only receives `(event, values)` from `listeners.notify`. So you must either:

- Pass the tree (or a small callback) into `PiGalleryElem` so it can call `_visible_rows` directly, or  
- Mirror the flag on **`pi_config`** (or similar) when the user toggles **Tree Score &lt; / &gt;=**, and read it in shared `visible_rows` logic.

This is the main “wiring” cost.

### 3. Order of operations in `PiGalleryElem.files_selected`

Conceptually:

```text
visible = same_set_as_tree_uses_for_file_nodes(values)   # was: c.table.rows()
self._base_collection_rows = SelectedTreeNodesFilter(files_folders).filter(visible)
self._apply_musiq_filter(values)   # gallery slider; unchanged
```

Using **`SelectedTreeNodesFilter(c.table.rows())` first and then intersecting with `visible`** is equivalent; the important part is that the operand to the tree filter is **`c.table.rows()`** and the result is what you intersect with folder selection.

### 4. Edge cases to keep in mind

- **Listener `values`**: Sometimes partial dicts are merged (`_musiq_threshold` already merges with `c.last_window_values`). Tree score should use the same pattern so the threshold is consistent when `-TREE-` fires.
- **Rebuild paths**: When the tree rebuilds after load/score filter, it calls **`_notify_tree_selection_with_threshold`**, which notifies listeners with **merged** values including **`values['-TREE-']`** set from **`_current_tree_selection_keys()`**. The gallery still gets a coherent `values` snapshot for sliders if you merge like the tree does.

## Summary

To make the gallery match tree folder contents **before** the gallery slider:

1. **Stop** basing gallery folder filtering on **`c.table.rows()`** alone; **base it on the same visible row set the tree uses** (`_visible_rows` semantics).
2. **Solve** where **`_score_cmp_less_than`** is read from (tree ref vs shared config vs extracted function taking an explicit flag).
3. Keep **`SelectedTreeNodesFilter`** as-is on that visible list; leave **`_apply_musiq_filter`** as the only gallery-specific narrowing after that.

Rough effort: small refactor (extract + wire comparator visibility) plus one call-site change in **`PiGalleryElem.files_selected`** and **`main`** if you inject the tree; duplicates tab (`PiDupElem`) may deserve the same treatment if it should stay consistent with the tree.

**Cursor**

Yes, **it can simplify things**, mainly in **behavior and duplication**, not necessarily in **line count** at first.

## What gets simpler

**One notion of “candidate files.”** Right now you effectively maintain two stories:

- Tree: `Show filter` + tree MusIQ (+ &lt;/&gt;=).
- Gallery base: `Show filter` only, then folder selection, then gallery MusIQ.

Aligning the gallery base with `_visible_rows` collapses that to: **everything downstream only disagrees on the gallery MusIQ slider**, not on “which rows exist for this score mode at all.” Fewer surprises, fewer “why is this in the gallery but not in the tree?” cases.

**Less duplicated MusIQ semantics.** Today tree and gallery each encode “what counts as passing score” (different thresholds, **&gt;=** vs **strict &gt;**, plus tree-only **&lt;** mode). Feeding the gallery from one **`visible_rows`** (or one shared helper) means **folder membership vs tree listing** stays consistent without re-explaining rules in comments.

**Clearer pipeline.** Mentally it becomes:

`table.rows()` → **tree visibility** → **selection / folder filter** → **gallery-only MusIQ**

instead of two parallel pipelines that almost match.

## What does not simplify much (or adds a bit of wiring)

You still need **two sliders** if you want gallery finer control than the tree, unless you remove the gallery slider later.

You must **expose tree visibility** without spreading **three** copies of the same logic: that usually means a **small shared function** or **passing the tree / a callback** into the gallery. That is a modest structural change, not a big framework.

## Bottom line

- **Conceptually and for maintenance:** yes, it usually **simplifies** the model and shrinks the space of bugs and edge cases.  
- **Mechanically:** you trade a thin **integration** layer (shared `visible_rows` + comparator visibility) for **no longer** debugging two divergent bases. Net simplification for anyone reading or changing the code is often real, even if `main` or `pi_config` gains a line or two of wiring.
