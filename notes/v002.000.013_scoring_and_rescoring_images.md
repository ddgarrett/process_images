```472:490:/Users/douglasgarrett/Documents/projects/image_analysis_lib/image_analysis_lib/duplicates.py
    duplicate_to_keeper: dict[str, str],
```

## 2) How dedup decides `status = "dup"` (cosine logic)

`duplicate_to_keeper` is built in `find_duplicates_by_score`:

- Comparisons are within same folder only.
- Optional GPS gate can skip comparisons if distance exceeds radius.
- If cosine similarity `>= min_similarity_threshold` (default `0.65`), lower-scoring image becomes duplicate of higher-scoring keeper.

## Accuracy limits vs original dedup

Your "quick and dirty" idea is sound, but here is where it can diverge from true original dedup:

- **If you do not recompute embeddings**: changing cosine threshold cannot be applied accurately.
- **If you recalc only one folder subset**: keeper/duplicate outcomes can differ if original run had a broader candidate set (even though current dedup buckets by same parent folder, data presence still matters).
- **File set drift**: added/removed images since dedup run changes ranking and duplicate links.
- **MUSIQ drift**: if `musiq_score` values came from different model/size/run than parms imply, outcome differs.
- **GPS data mismatch**: missing/changed EXIF GPS in collection vs original file read path changes gating results.
- **Tie handling**: equal scores rely on ordering details; small differences can change keeper choice.
- **Manual edits in collection**: user-changed `img_status`/`rvw_lvl`/`dup_photo` will be overwritten or conflict with recalculated values.
- **Parent fallback parms**: using parent `dedup_parms.json` for child folder is convenient, but may not match how that child was originally run.

Here is how the plan reads with your additions, plus GPS confirmation and follow-on concerns.

## GPS: collection `img_lat` / `img_lon` vs dedup

They are **the same kind of EXIF GPS** (DMS from `exifread`, latitude/longitude ref applied), but they are **not guaranteed bit-identical** in all cases.

- **Dedup** reads GPS **from the image file on disk** when it runs, via `get_gps_from_exif()` in `image_analysis_lib` (used by `build_gps_cache()` in the duplicate finder and when writing `image_scores_and_status.csv`).
- **Collection** fills `img_lat` / `img_lon` in `ExifLoader` using `_get_lat_lon()` on the same GPS tags (`GPS GPSLatitude` + ref, `GPS GPSLongitude` + ref in `image_collection_metadata.csv`), and **rounds to 6 decimal places**.

So: **same EXIF fields and same general formula**; dedup uses full float from the parser for distance, while the collection stores **rounded** values. They can also diverge if the **file or EXIF changes** after the collection row was built, or if one path reads a file that was moved/corrupt.

Writing **only** next to `image_scores_and_status.csv` (the `image_root` / day directory passed to dedupe) matches how `cli.py` already treats that directory as the single output root for the status CSV. Your manual copy/edit workflow fits that.

For **latitude**, your intuition is right: one **sixth decimal place** of a degree is about **0.11 m** (111 km per degree / 10^6 ≈ 0.111 m per 0.000001°).

For **longitude**, the distance per degree **shrinks** as you move away from the equator by about `cos(latitude)`. So at latitude 45°, one unit in the sixth decimal place of **longitude** is roughly **0.11 m × cos(45°) ≈ 0.08 m** still in the same ballpark.

So **6 decimal places** is **well below** typical GPS consumer noise and below the dedup **GPS radius** values people use (often tens to hundreds of meters). Rounding in the collection versus full float in dedup should not matter for your duplicate gating in practice.

```bash
pip install --no-cache-dir \

```

```bash
pip install --no-cache-dir -r requirements.txt
```

```bash
pip show image-analysis-lib
```
