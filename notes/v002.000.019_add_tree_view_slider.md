**Cursor**

Great question. The loop risk comes from this pattern:

1. User moves slider A.
2. Handler for slider A runs.
3. Inside that handler, code programmatically updates slider B (or A itself).
4. That update fires slider B's handler.
5. Slider B handler then updates slider A again.
6. Repeat.

Even if FreeSimpleGUI sometimes suppresses events for programmatic updates, you do not want to rely on that behavior across platforms and versions.

Simple example of the bad pattern:

```python
def on_tree_slider(event, values):
    new_v = values[TREE_SLIDER]
    window[GALLERY_SLIDER].update(value=new_v)  # may fire gallery handler

def on_gallery_slider(event, values):
    new_v = values[GALLERY_SLIDER]
    window[TREE_SLIDER].update(value=new_v)     # may fire tree handler
```

If both fire each other, you get recursive churn, UI lag, or repeated rebuilds.

Use a guard:

```python
self._syncing_sliders = True
try:
    window[GALLERY_SLIDER].update(value=v)
    window[DUP_SLIDER].update(value=v)
finally:
    self._syncing_sliders = False
```

And at top of each slider handler:

```python
if self._syncing_sliders:
    return
```

So the problem scenario is "handler updates another control that has its own handler, and that handler updates back." The guard makes those programmatic sync updates one-way and non-reentrant.

## Recommended pattern

Before rebuilding tree data, capture UI state from the tree widget:

- selected item keys
- expanded folder keys
- optionally current focus key

Rebuild the tree with new filtered rows, then restore only keys that still exist.

Because filtering removes some nodes, restore is always "intersection with current keys".

## Why it works here

Your tree already has key mapping support after update:

```110:115:pi_tree_list.py
    def update_list(self,event,values):
        self._tree_data = PiTreeData(c.table.rows())
        c.window[self.key].update(values=self._tree_data)
        self._key_id_dict = {v:k for k, v in self._tree.IdToKey.items()}
```

That means after each rebuild you can check which previous keys still exist and restore state for those only.

## Practical restore rules

- **Selection:** restore only selected keys still present.
- **Expansion:** restore only folder keys still present.
- **Focus:** restore if still present, else pick first selected or leave unset.

For `Show...` events, use the exact same function you use for tree-slider refresh:
1) snapshot state
2) rebuild with new rows
3) restore state by key intersection

So yes, one shared helper can serve both paths cleanly.
