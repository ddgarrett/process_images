## Important edge cases / decisions to confirm

These are the main questions I would want answered before implementation:

- **Duplicate identity key**: should duplicate mean `(file_location, file_name)` only? (That is what the app commonly uses.)
- **Path normalization**: should comparisons normalize slashes, case, and leading `/` so `\a\b` and `/a/b` match?
- **Copy collision policy** when target file already exists on disk:
  - overwrite,
  - skip,
  - or compare size/hash then decide?
- **Missing source image file** for a CSV row: skip and log error, or abort entire merge?
- **Atomicity**:
  - continue-on-error and report all issues (usually best UX), or
  - fail-fast and rollback (harder because files may already be copied)?
- **CSV ordering/IDs**:
  - run existing `resort()` and `renumber()` after merge,
  - or preserve incoming `file_id` values?
- **Filtered table state**: if a filter is active, merge should operate on unfiltered `_original_rows` to avoid accidental misses.
- **Hidden/system folders**: should merge honor same skip rules as loader (`_`, `.`, `$`), or trust CSV fully?
- **Encoding/BOM**: source CSV may be UTF-8 with BOM; should be read with `utf-8-sig` for safety.
- **Column mismatch strictness**:
  - exact order + exact names required (as your request implies),
  - or allow same set in different order?
- **UI surface**:
  - add as File menu action like "Merge Collection...",
  - and whether to show per-file errors only in console or also status popup.
