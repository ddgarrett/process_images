The same event is also on the main **File** menu (`pi_menu.py`), not only the right-click menu.

## What happens when you choose it

`EVT_FILE_PROPS` is registered to `PiFileProperties` in `pi_config.py`:

```107:112:pi_config.py
''' standard global actions '''
PiOpenCollection(EVT_FILE_OPEN)
PiNewCollection(EVT_FILE_NEW)
PiAddFolders(EVT_ADD_FOLDERS)
PiSaveCollection(EVT_FILE_SAVE)
PiFileProperties(EVT_FILE_PROPS)
```

The handler is a small debug-style dialog: it opens a second window titled **Second Form**, shows the event name and the full `values` dict as text, waits for you to click **OK**, then closes that window:

```7:14:pi_action_props.py
class PiFileProperties(PiAction):
    def handle_event(self,event,values):
        layout = [[sg.Text(f'Properties Window \nEvent: {event}\nValues: {values}')],
                 [sg.OK()]]

        window = sg.Window('Second Form', layout)
        window.read() # wait for any event
        window.close()
```

So in practice, **Properties** does not show or edit real file or image metadata yet; the module comment says it should *display and update* properties, but the implementation is only a placeholder that dumps the GUI event and `values` for debugging.

Overall this is **moderate effort**: a focused feature, but a few integration points in this codebase need care so it feels reliable rather than "works on my machine."

## What is already easy

- **`img_lat` / `img_lon` / `file_name`** already exist on each collection row and are defined in `image_collection_metadata.csv` (see rows 4-7 in that file). Showing them and binding inputs is straightforward.
- **Thumbnails** reuse existing helpers: `pi_image_util.cnv_image` can load from the full path (`c.directory` + `get_fn_for_row(row)` from `pi_util.py`) and resize for a small preview.
- **Save / Cancel** in a modal `sg.Window` is standard FreeSimpleGUI: loop with `read()`, on Save validate then write into the row object(s), on Cancel just `close()`.

## What adds real work

**1. New columns are not in the data model yet**

`image_caption`, `image_intro_paragraph`, and `image_album_uri` do not appear in `image_collection_metadata.csv`. The blog generator (`pi_action_blog.py`) only emits literal placeholders like `{image_intro_paragraph}` in the HTML; it does not read those from the row today.

You would append new metadata rows (and defaults, typically empty strings). `csv_table.py` is built so **extra metadata columns at the end** extend each row on load when older CSVs have fewer columns, which matches how you would roll this out.

**2. Picking exactly one row for "this image"**

`PiFileProperties` is a single global handler for `EVT_FILE_PROPS`. Right-click already sets `c.status_menu_rowgetter` before the menu opens (see `pi_macos_popup._bind_popup_to_widget` and `pi_status_menu_context.py`), which is the same pattern status actions use.

You still need clear rules:

- If the rowgetter returns **multiple** rows (tree selection can include a whole folder), either show an error, take the first image only, or disable Properties unless exactly one file row is selected.
- From the **File** menu (not right-click), `status_menu_rowgetter` may be **stale** or wrong; you need a fallback using tree selection from `values` or `c.last_window_values`. On macOS, popup items can fire with `values` as `None` (`write_event_value(..., None)`), so `c.last_window_values` matters there too (already noted in `pi_config.py`).

**3. Thumbnail lifetime**

Tk `PhotoImage` objects used in `sg.Image` should be **kept referenced** (for example on the dialog window object) until the window closes, or the preview can go blank when Python GC runs.

**4. What "Save" should do**

Rows are normal in-memory row objects; persistence is `c.table.save()` (see `PiSaveCollection` in `pi_action.py`). You can either:

- Save only to the in-memory row and let the user use **File > Save** for CSV, or  
- On dialog **Save**, write the row and call `c.table.save()` immediately.

You also likely want `c.listeners.notify(c.EVT_TABLE_ROW_CHG, ...)` after updating fields so the tree/gallery refresh pattern stays consistent with other edits (same idea as `pi_status_apply.py`).

**5. Validation**

Parse latitude/longitude as floats, handle blank, and decide whether to allow writing back to EXIF or **only** to the CSV (your question implies CSV fields, which is simpler).

## Bottom line

For someone comfortable with this app, expect on the order of **roughly half a day to a day** to get a solid first version (schema + dialog + row resolution + thumbnail + persist + refresh), and a bit more if you want strict single-selection behavior, polished validation, and thorough testing across tree vs image vs gallery vs File menu and macOS vs Linux.

None of it is exotic; the main risk is **row selection and `values` on macOS**, not the form layout itself.

Your decisions line up with the current code. Here are precise notes and the remaining open issues.

## 1. `pi_action_blog.py` and placeholders

`get_body` is an **f-string**, so what you write as `{{image_intro_paragraph}}` in Python becomes a **single-brace** token in the written HTML: `{image_intro_paragraph}`. Same for `{album}` and `{image_caption}`. So the rule you want is: **if the row field is non-blank after strip, emit that value; else emit the same `{...}` token that is emitted today** (so the skeleton stays editable by hand).

`get_footer` is a normal string (not an f-string), so it already contains literal `{album}` and `{album_name}`. For consistency you will probably want **the same substitution rule for `{album}` in the footer** when `image_album_uri` is non-blank, and leave `{album}` as-is when blank. `{album_name}` has no column yet unless you add one; leaving it as a manual placeholder is fine unless you want a dedicated field later.

`{{image_extended_description}}` in `get_body` becomes `{image_extended_description}` in the file. You did not define a CSV field for it; leaving that placeholder unchanged unless you add a column later is consistent.

**HTML escaping:** If you inject caption, description, or album text into HTML, raw `<` and `>` in user text can break markup or create XSS if the HTML is ever served. Escaping on output (or storing plain text only and escaping at blog generation) is a small extra decision.

## 2. First row only; remove File > Properties

Using **only the first row** when the rowgetter returns several is straightforward.

Removing **File > Properties** means editing `pi_menu.py` (the line with `EVT_FILE_PROPS` under `&File`). Right-click **Properties** on tree, image, and gallery stays as today (`pi_tree_list.py`, `pi_image_elem.py`, `pi_gallery_elem.py`). No `Properties` entry showed up under `pi_oi_menu.py` in a quick search.

## 3. Thumbnail reference

Keeping the `PhotoImage` (or whatever `cnv_image` returns) on the dialog object for the lifetime of the window is the right approach.

## 4. "Update" and in-memory only

Renaming the button to **Update** and only mutating the row in `c.table` matches how the rest of the app treats persistence (`PiSaveCollection` calls `c.table.save()`). You may still want to fire **`EVT_TABLE_ROW_CHG`** (with the updated row(s)) after a successful Update so any UI that depends on row data stays in sync, without writing the CSV.

## 5. Validation

- **GPS:** After strip, empty means clear or leave as empty in the row (your "blanks stay blank"). Non-blank should parse as floats with **latitude in [-90, 90]** and **longitude in [-180, 180]** (and decide how strict to be about `0` and tiny floats from old data).

- **URL "returns something":** This implies a **network request** (for example `HEAD` or a small `GET`) with a **timeout**, optional **redirect** handling, and a **2xx** (or sometimes 3xx) success rule you define. Open points:
  - Some hosts answer `GET` but not `HEAD`.
  - TLS errors, captchas, and bot blocking can make a valid browser URL "fail" from the app.
  - Running this on the UI thread will **freeze the dialog** until the request finishes; a short timeout (for example a few seconds) limits damage but can still feel sluggish on bad networks.

You said you are **not** updating EXIF; only CSV-backed row fields in memory. That stays simple.

## Other open issues worth deciding up front

1. **Column names vs blog tokens:** Store as `image_caption`, `image_intro_paragraph`, `image_album_uri` in metadata/CSV while substituting into `{image_caption}`, `{image_intro_paragraph}`, and `{album}` in the HTML. No conflict, just keep the mapping explicit in one helper.

2. **`get_header()` uses `{intro}`:** That is unrelated to the new fields unless you later tie intro to a row or global; no change required for your current scope.

3. **macOS and `values`:** Properties from the Tk popup path can still arrive with **`values` is `None`**; resolving the row from **`c.status_menu_rowgetter(c.last_window_values)`** (with the "first of list" rule) remains important. Removing the File menu item reduces bad context from "File > Properties" but right-click-only is actually cleaner for row identity.

4. **Blog generation and URL check:** Blog export can **trust** fields already validated at Update time, or re-check URLs when generating the blog (slower, redundant). Your call.

5. **Relative vs absolute album URLs:** `urllib` validation usually needs a **scheme** (`https://...`). If users omit `https://`, you may want to normalize or show a clear validation message.

6. **Row types from CSV:** Loaded rows may have numbers as strings; normalization on Update (store float for lat/lon or consistent string) avoids odd comparisons later.

None of these block the design; they are the usual polish items so behavior matches user expectations across platforms and bad inputs.
