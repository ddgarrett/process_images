Overall this is **moderate effort**: a focused feature, but a few integration points in this codebase need care so it feels reliable rather than "works on my machine."

## What is already easy

- **`img_lat` / `img_lon` / `file_name`** already exist on each collection row and are defined in `image_collection_metadata.csv` (see rows 4-7 in that file). Showing them and binding inputs is straightforward.
- **Thumbnails** reuse existing helpers: `pi_image_util.cnv_image` can load from the full path (`c.directory` + `get_fn_for_row(row)` from `pi_util.py`) and resize for a small preview.
- **Save / Cancel** in a modal `sg.Window` is standard FreeSimpleGUI: loop with `read()`, on Save validate then write into the row object(s), on Cancel just `close()`.

## What adds real work

**1. New columns are not in the data model yet**

`image_caption`, `image_intro_paragraph`, and `image_album_uri` do not appear in `image_collection_metadata.csv`. The blog generator (`pi_action_blog.py`) only emits literal placeholders like `{image_intro_paragraph}` in the HTML; it does not read those from the row today.

You would append new metadata rows (and defaults, typically empty strings). `csv_table.py` is built so **extra metadata columns at the end** extend each row on load when older CSVs have fewer columns, which matches how you would roll this out.

**2. Picking exactly one row for "this image"**

`PiFileProperties` is a single global handler for `EVT_FILE_PROPS`. Right-click already sets `c.status_menu_rowgetter` before the menu opens (see `pi_macos_popup._bind_popup_to_widget` and `pi_status_menu_context.py`), which is the same pattern status actions use.

You still need clear rules:

- If the rowgetter returns **multiple** rows (tree selection can include a whole folder), either show an error, take the first image only, or disable Properties unless exactly one file row is selected.
- From the **File** menu (not right-click), `status_menu_rowgetter` may be **stale** or wrong; you need a fallback using tree selection from `values` or `c.last_window_values`. On macOS, popup items can fire with `values` as `None` (`write_event_value(..., None)`), so `c.last_window_values` matters there too (already noted in `pi_config.py`).

**3. Thumbnail lifetime**

Tk `PhotoImage` objects used in `sg.Image` should be **kept referenced** (for example on the dialog window object) until the window closes, or the preview can go blank when Python GC runs.

**4. What "Save" should do**

Rows are normal in-memory row objects; persistence is `c.table.save()` (see `PiSaveCollection` in `pi_action.py`). You can either:

- Save only to the in-memory row and let the user use **File > Save** for CSV, or  
- On dialog **Save**, write the row and call `c.table.save()` immediately.

You also likely want `c.listeners.notify(c.EVT_TABLE_ROW_CHG, ...)` after updating fields so the tree/gallery refresh pattern stays consistent with other edits (same idea as `pi_status_apply.py`).

**5. Validation**

Parse latitude/longitude as floats, handle blank, and decide whether to allow writing back to EXIF or **only** to the CSV (your question implies CSV fields, which is simpler).

## 5. Validation

- **GPS:** After strip, empty means clear or leave as empty in the row (your "blanks stay blank"). Non-blank should parse as floats with **latitude in [-90, 90]** and **longitude in [-180, 180]** (and decide how strict to be about `0` and tiny floats from old data).

- **URL "returns something":** This implies a **network request** (for example `HEAD` or a small `GET`) with a **timeout**, optional **redirect** handling, and a **2xx** (or sometimes 3xx) success rule you define. Open points:
  - Some hosts answer `GET` but not `HEAD`.
  - TLS errors, captchas, and bot blocking can make a valid browser URL "fail" from the app.
  - Running this on the UI thread will **freeze the dialog** until the request finishes; a short timeout (for example a few seconds) limits damage but can still feel sluggish on bad networks.

You said you are **not** updating EXIF; only CSV-backed row fields in memory. That stays simple.

Here is a direct answer tied to your numbered points.

## 1 and 2 (CSVs and dual support)

If no CSVs were ever written with the old `image_*` names, you do not need migration or dual placeholder support. Single naming scheme (`img_*` everywhere) is enough.

## 3 (placement)

Keeping `img_ext_descr` in the same spot as today’s `{image_extended_description}` in `get_body` matches what you described earlier.

## 5 (length: unrestricted vs ~1024 bytes)

- **In memory / Python:** Unbounded strings are fine; a few KB is trivial.
- **CSV:** Long fields are OK if the row is written with normal CSV quoting (newlines inside a field require quotes). If you forbid real newlines in these fields (see below), quoting stays simple.
- **GUI:** Tk `Text` is fine for at least thousands of characters; 1024 bytes is not a real limit.

So “unrestricted” is usually safe here. If you ever add a cap, **1024 bytes** is a reasonable soft limit for captions/descriptions and keeps files small; it is not required for correctness.

## 4 (Enter, Tab, and word wrap) — difficulty and behavior

**Word wrap at spaces:** Tk’s `Text` widget supports `wrap='word'` (wrap between words). FreeSimpleGUI’s `Multiline` is backed by that widget, so you get visual wrapping when the widget has a bounded width (your `size` / `expand_x`). That part is standard.

**Enter should not insert a new line:** That is **not** the default for `Multiline`. Default is: Enter adds `\n`. To change it you hook the underlying Tk widget after `finalize=True`, e.g. `window['-KEY-'].Widget`, and bind `<Return>` / `<KP_Enter>` to a handler that returns `'break'` so the default insert does not run.

**Tab should not insert a tab:** Similarly, bind `<Tab>` on that widget to return `'break'` (and optionally move focus to the next field with `tk_focusNext` if you want normal form behavior).

**Difficulty:** **Low to moderate** — a small amount of platform-specific Tk code, not a large refactor. The awkward part is testing on macOS (your app already uses Tk) and making sure bindings do not fight FreeSimpleGUI’s own focus handling.

**Questions / edge cases this raises:**

1. **Paste:** Blocking Enter stops typed newlines, but paste can still insert `\n` and `\t`. If you want strict “no newlines / no tabs,” strip or replace those on **Update** (or on `<<Paste>>`) as well as key bindings.
2. **Copy/paste from web:** Rich paste can bring odd characters; stripping `\r`, normalizing line endings, or replacing runs of whitespace might be a separate choice.
3. **Accessibility:** Turning Enter into “no newline” is unusual for a tall text box; users might still expect Shift+Enter or paste for paragraphs. Your spec is clear; just document it in the UI (e.g. hint text) if needed.
4. **Two fields:** You will likely duplicate the same bind logic for `img_intro_paragraph` and `img_ext_descr`, or factor a tiny helper that configures a `Multiline` widget once.

So: **visual word wrap** is easy; **no Enter / no Tab** is a short Tk binding layer plus optional paste/update sanitization if you want it bulletproof.

- Replace any run of whitespace (newlines, tabs, carriage returns, multiple spaces) with **a single ASCII space**, then `.strip()` ends.

- `re.sub(r'\s+', ' ', text).strip()`  
  or split/join: `' '.join(text.split())`

That matches “Enter and Tab become spaces” and also collapses accidental double spaces.

**Things to be aware of**

1. **Paragraph structure is lost** on save. If that is OK for blog snippets and CSV, fine.
2. **Show the user what will be stored** (optional): you could preview normalized text, or keep it invisible until they reopen the dialog.
3. **CSV**: A single line of text with only spaces is easy to quote; no embedded newlines in the field unless you later allow them again.
4. **Display**: After reload, the string is one long line; `Multiline` still **wraps visually** if `wrap='word'` and width is set, so it still reads like a paragraph.

**Verdict**

Allow Enter/Tab in the UI, normalize on **Update** with `\s+` → single space. Low complexity, predictable behavior, offline-friendly. No new technical blockers beyond deciding whether to normalize on every keystroke (usually no) or only on Update (yes).
