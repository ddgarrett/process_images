The **tree** passes `get_selected_rows`, which uses the current tree selection (`values` for the tree key) and `SelectedTreeNodesFilter`, so **every row that matches the selected tree nodes** (files and whole folders) gets included:

```64:79:pi_tree_list.py
    def get_selected_rows(self,values):
        ''' Return a list of selected rows. 
            A selected folder returns all of the rows in the folder.
            Values is the values returned by window.read().
            Therefore we extract the list of selected
            files and folders via our own key.
        '''
        files_folders = values[self.key]
        rows = self._tree_data.rows
        if len(rows) == 0 or len(files_folders) == 0:
            return []
        
        filter = SelectedTreeNodesFilter(files_folders)
        rows = filter.filter(rows)
        # print(f'{len(rows)} rows selected')
        return rows
```

The **gallery** passes `get_rows`, which **does not use `values` at all**; it only uses thumbnail selection:

```113:115:pi_gallery_elem.py
    def get_rows(self,values):
        ''' Used by StatusMenu items and PiActionMap to get selected row '''
        return self._selected_rows
```

So if you see **all tree-selected images** change status, the row list is coming from **`get_selected_rows` (tree semantics)**. With the current gallery API, that should only happen if the **handler that runs is the one attached to the tree’s `StatusMenuItem` instances**, not the gallery’s (because gallery’s handler would only ever iterate `self._selected_rows`).

## Most plausible causes

1. **Wrong menu / wrong event for this click**  
   The GUI stack may sometimes associate the click with the **tree’s** context menu (same labels, but different embedded event IDs), so the event that reaches `main` is still a tree `-STATUSITEM_n-` handler wired to `get_selected_rows`. That matches “sometimes” and matches the symptom exactly.

2. **macOS dual path**  
   On Darwin, `install_macos_popup_fallbacks` adds Tk popups built from `gallery._menu`, while FreeSimpleGUI may still do its own right-click handling. In theory two layers could interact oddly, though the usual risk is duplicate events rather than silently swapping menus; still worth mentally isolating (you’d have to try with/without the fallback in Agent mode).

3. **Less likely: `_selected_rows` equals “everything”**  
   From the code, `_selected_rows` is only filled in `thumb_selected` and cleared on tree/filter/page changes. There is no path that sets `_selected_rows` to the full tree-backed collection unless you actually toggled every thumb. So this usually **does not** explain “whole tree selection” by itself.

Here is what would change in practice if you moved to **one logical “Set status” menu** instead of **many copies** (tree, gallery, image, duplicates each building their own `StatusMenu(...)` and their own `StatusMenuItem` instances).

## What you have today

Each pane creates its own `StatusMenu(rowget=...)`, which instantiates its own block of `StatusMenuItem`s. Each item:

- Registers a **global** listener on a **unique** event id (`-STATUSITEM_n-`).
- Bakes in **one** `rowget` (tree vs gallery vs image).

So “the same” user-facing menu is really **several parallel menus** wired to different handlers and ids.

## What would change with a single logical menu

**1. Event routing**

Instead of many `-STATUSITEM_n-` listeners, you typically have:

- **One** small set of events (or one dispatch event), and  
- **One** handler that decides **which rows to touch** from **context**, not from which copy of `StatusMenuItem` fired.

Context might be: last right-click target (tree vs thumbnail index vs single image), focus, or a flag set when opening the menu.

**2. Listener / registration model**

- **Fewer** `c.listeners.add(...)` entries for status actions.
- **No** dependence on construction order for id ranges (tree before gallery, etc.).
- **Lower** risk of “wrong” global handler running if the toolkit reused or mixed up menu instances (the class of bug you were debugging).

**3. Where “which rows?” is decided**

Today that is split: each menu copy hard-wires `rowget`. With one menu, you’d centralize something like:

- If source is **gallery**: rows = gallery’s `_selected_rows` (or “right-clicked thumb only,” if you add that rule).
- If source is **tree**: rows = `get_selected_rows(values)`.
- If source is **image pane**: rows = `[that one row]`.

So **one** place encodes the rules instead of **N** copies of `StatusMenu` each carrying a different `rowget`.

**4. Menu definitions (layout code)**

`PiTreeList`, `PiGalleryElem`, `PiImageElem`, (and dup) would **not** each call `StatusMenu(...).get_set_menu()` in their `__init__`. They might still pass **the same menu structure** into `right_click_menu=`, but that structure would reference **shared** event strings, or a **non-item-specific** submenu built once (e.g. at startup).

**5. macOS Tk fallback (`pi_macos_popup.py`)**

Today each widget gets a popup built from **its** `element._menu`. With one shared definition:

- You’d build **one** Tk popup from the **canonical** menu def, or  
- Keep **per-widget** popups but ensure they all use the **same** command strings and that **opening** the menu sets context so the single handler behaves correctly.

Either way, the **commands** would no longer need to differ per pane just to bind a different `rowget`.

**6. Behavior you can define more clearly**

You could standardize rules that are fuzzy today, for example:

- Right-click on a gallery thumb: status applies to **only that thumb**, or **all gallery-selected thumbs**, regardless of tree selection.
- Tree context menu always uses tree selection.

That becomes explicit in **one** dispatch function instead of implicit in **which menu copy** was constructed.

**7. Testing and mental model**

- Easier to reason about: “status change always goes through `apply_status(context, status, level)`.”
- Easier to test: one entry point, mock `context` / `values`.

## Tradeoffs

- **More plumbing up front**: you need a reliable **context** (who opened the menu) and to keep it in sync with `window.read()` / `values`.
- **Slightly more state** (e.g. `last_popup_source`) unless the GUI gives you enough in the event payload.
- **Less duplication** in menu definitions and fewer moving parts for duplicate listeners and ids.

**Bottom line:** the main change is **not** the on-screen labels; it is shifting from **many globally registered handlers, each tied to a different `rowget`**, to **fewer events + explicit context** so “same menu” always means one code path that chooses rows deliberately. That usually reduces spooky cross-pane bugs at the cost of a bit of explicit state or routing logic.

## Summary

### 1. **`pi_config.py`**
- Added **11 fixed event names** (`EVT_STATUS_SET_REJECT`, `EVT_STATUS_TBD_GOOD_OR_BEST`, etc.).
- Added **`status_menu_rowgetter`**: `callable(values) -> rows`, set on right-click before the menu is used.
- Added **`last_window_values`**: updated each `window.read()` so macOS Tk `write_event_value(..., None)` still has a values dict for tree `get_selected_rows`.
- **Bugfix:** `PiFilterTable(c.EVT_SHOW_...)` was invalid (`c` not defined in that module); switched to `EVT_SHOW_ALL`, etc.

### 2. **`pi_status_apply.py`** (new)
- **`register_status_menu_handlers()`** (called once from `main()`): registers **11** `PiAction` handlers, one per canonical event.
- Each handler reads **`c.status_menu_rowgetter`**, resolves rows (drops `None`), updates `rvw_lvl` / `img_status`, then notifies **`EVT_TABLE_ROW_CHG`**.
- **`_effective_values`**: uses `last_window_values` when the event carries `values is None` (macOS popup path).

### 3. **`status_menu.py`**
- **`get_set_menu()`** is now a **`@staticmethod`** that returns only **strings** like `f'&Reject::{c.EVT_STATUS_SET_REJECT}'` (no `StatusMenuItem` instances, no extra listeners).
- **`StatusMenu(rowget)`** is unchanged for **`get_show_submenu()`** / duplicate-group actions that still need a row getter.

### 4. **`pi_macos_popup.py`**
- **`_bind_popup_to_widget(..., rowgetter=...)`** sets **`c.status_menu_rowgetter`** at the start of **`_show_popup`** (before **`tk_popup`**), with the right method per widget: tree / image / gallery / dup.

### 5. **`pi_status_menu_context.py`** (new)
- On **non-Darwin**, binds **`Button-3` / `Button-2` / `Control-Button-1`** on the same widgets so **`status_menu_rowgetter`** is set before FreeSimpleGUI’s own context menu (Linux/Pi/Windows).

### 6. **`main.py`**
- Calls **`register_status_menu_handlers()`** before **`init_window()`**.
- After popups: **`install_status_menu_context_bindings(c.window, _ui_refs)`**.
- Updates **`c.last_window_values`** whenever **`values`** from **`read()`** is not `None`.

### 7. **`status_menu_item.py`**
- Reduced to a short comment: logic lives in **`pi_status_apply`** (avoids keeping a dead `PiAction` subclass).

The refactor **does not** change whether FreeSimpleGUI’s built-in right-click path works on Darwin. The workaround would only become “unnecessary” if **Apple + Tk + FreeSimpleGUI** behavior improved so `right_click_menu` **always** works on your Mac **without** extra binds. The centralized status refactor does not cause that.

**Bottom line:** keep the macOS workaround unless you confirm native right-click menus work reliably on your target Macs without it; the status-menu centralization is still worth doing for correctness of **which rows** change when the menu **does** fire.
