**Yes, for the media CSV.** If a `--paths` entry does not exist on disk, you still get a warning and the walk does nothing for that root, but the merge logic **drops** prior rows that live under that root and **does not** add replacements.

What happens:

1. **`kept_media`** keeps only existing rows whose `Full Path` is **not** under any normalized `--paths` value:

```310:314:reconcile_media.py
        kept_media = [
            r
            for r in all_existing
            if not row_matches_any_scan_root(r["Full Path"], scan_roots_norm)
        ]
```

2. **Missing root:** `scan_media_directories` skips it and prints a warning, so **`new_media` has no rows** for that tree:

```160:163:reconcile_media.py

        if not os.path.exists(root_path):
            print(f"Warning: Path does not exist -> {root_path}")
            continue
```

3. **`merged_media = kept_media + new_media`** therefore **omits** every old row that was under the missing path.

So those entries are removed from the output file on the next write, not because of an explicit “delete missing path” step, but because they are excluded from `kept_media` and never re-scanned.

**Parms CSV:** `merge_parm_rows` removes prior parm rows whose `Path` is in this run’s `scan_paths_norm` and appends one new row per `--paths` entry (including the missing one), so you still get a parms line for that path with the new run date/time.

**Edge case:** If stored `Full Path` values in the CSV **do not** match what `norm_path` / `path_is_under` considers “under” the `--paths` you pass (different spelling, symlink resolution, etc.), a row might incorrectly stay in `kept_media` or incorrectly be dropped.
